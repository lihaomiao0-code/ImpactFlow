# PRD｜AI 代码变更影响分析平台

## 1. 文档信息
- 产品名称：ImpactFlow
- 文档版本：V1.0
- 文档类型：产品需求文档（PRD）
- 目标形态：研发效能工具 / 代码变更影响分析平台
- 目标平台：Web 应用，支持私有化部署
- 当前阶段：MVP 需求定义

## 2. 背景与问题
在研发协作中，代码变更常常带来以下问题：
1. 变更影响范围不明确，开发者难以快速判断改动会波及哪些模块。
2. Code Review 依赖人工经验，审查效率低且容易遗漏风险点。
3. 测试团队难以及时识别需要回归的核心路径。
4. 发布前缺少统一的风险评估与影响分析能力。
5. 对复杂调用链、间接依赖、跨模块传播的识别成本高。

当前研发团队需要一种工具，能够在代码变更前后自动分析影响范围、识别潜在风险，并给出可执行的测试与修复建议。

## 3. 产品目标
### 3.1 核心目标
构建一个 AI 代码变更影响分析平台，能够自动识别代码改动带来的：
- 影响范围
- 调用链传播
- 风险等级
- 测试建议
- 修复建议

### 3.2 业务目标
- 提高 PR 审查效率
- 降低回归缺陷率
- 提前发现高风险变更
- 缩短分析复杂代码影响面的时间
- 为研发流程增加自动化质量门禁

### 3.3 成功指标
- PR 审查平均耗时下降
- 高风险变更提前识别率提升
- 回归缺陷数量下降
- 自动建议采纳率提升
- 分析结果可解释性评分提升

## 4. 产品定位
### 4.1 一句话描述
ImpactFlow 是一个面向研发团队的代码变更影响分析平台，能够自动识别改动、评估传播范围、输出风险等级，并给出测试与修复建议，帮助团队在 PR 和发布前降低回归风险。

### 4.2 产品价值
- 帮助开发者理解改动影响
- 帮助 Reviewer 快速判断风险
- 帮助测试确定回归重点
- 帮助技术负责人做发布决策

## 5. 目标用户
### 5.1 开发工程师
- 提交代码前确认变更影响
- 快速定位潜在回归点
- 获取补测与修复建议

### 5.2 Reviewer / Code Reviewer
- 快速理解 PR 风险
- 验证变更是否破坏调用契约
- 判断是否需要进一步审查

### 5.3 测试工程师
- 获取受影响模块清单
- 生成回归测试范围
- 补充边界条件与集成测试

### 5.4 技术负责人 / 架构师
- 评估核心链路风险
- 判断是否需要降级、灰度或回滚预案
- 进行跨模块影响分析

## 6. 使用场景
### 6.1 提交 PR 前
开发者上传变更或关联 diff，系统自动输出影响范围、风险等级和建议。

### 6.2 Code Review 期间
Reviewer 查看分析结果，快速判断变更是否会影响关键链路。

### 6.3 发布前检查
在上线前对所有变更执行影响分析，识别高风险改动。

### 6.4 事故回溯
在故障发生后，通过历史变更分析可能的影响传播路径。

## 8. 功能需求
### 8.1 仓库接入
- 支持上传 ZIP 仓库
- 支持输入本地仓库路径
- 支持手动编辑示例仓库
- 后续支持 Git 仓库地址

### 8.2 代码解析
- AST 解析
- 函数识别
- 模块识别
- 导入关系识别
- 调用关系识别
- 依赖图构建

### 8.3 变更识别
- 支持多文件 diff
- 支持局部变更定位
- 支持行级变更识别
- 对格式化或注释变更进行降噪处理

### 8.4 影响传播分析
- 上游依赖识别
- 下游影响传播
- 影响深度计算
- 间接依赖识别
- 高风险链路定位

### 8.5 风险评估
- 接口契约变更
- 参数变化
- 返回值变化
- 核心路径修改
- 高耦合模块修改
- 影响范围广
- 测试覆盖不足

### 8.6 修复建议生成
- 参数校验建议
- 空值保护建议
- 兼容性建议
- 接口封装建议
- 回滚策略建议

### 8.7 测试建议生成
- 单元测试
- 集成测试
- 边界条件测试
- 异常场景测试
- 回归测试
- 端到端测试

### 8.8 报告输出
- 仓库概览
- 变更摘要
- 影响范围
- 风险等级
- 修复建议
- 测试建议
- 备注说明

### 8.9 可视化展示
- 文件数
- 函数数
- 变更函数数
- 影响函数数
- 依赖边数
- 变更函数列表
- 受影响范围列表
- 关联建议

## 9. 非功能需求
### 9.1 性能
- 小型仓库应可快速完成分析
- 中型仓库应在可接受时间内给出结果
- 后续支持增量分析和缓存机制

### 9.2 准确性
- 函数识别准确
- 依赖关系准确
- 影响链召回率足够高
- 风险评估结果具有可解释性

### 9.3 可用性
- 页面交互简单
- 分析结果易于理解
- 失败提示明确
- 支持默认样例快速体验

### 9.4 安全性
- 仓库代码不外泄
- 上传文件应可隔离处理
- 支持私有化部署
- 后续增加敏感信息扫描

### 9.5 可扩展性
- 支持多语言接入
- 支持更多仓库来源
- 支持更多风险规则
- 支持 AI 增强能力

## 10. 核心流程
1. 导入仓库
2. 选择分析方式
3. 提交 diff
4. 系统解析代码结构
5. 系统识别变更点
6. 系统推导影响范围
7. 系统生成风险、测试和修复建议
8. 用户查看报告并导出结果

## 13. 风险与挑战
### 13.1 技术风险
- 大仓库解析性能问题
- 多语言 AST 解析复杂
- 变更与函数映射不准确
- 调用链识别误报或漏报

### 13.2 产品风险
- 用户不理解分析结果
- 风险等级缺乏可信度
- 建议过多导致信息噪音
- 与现有研发流程集成成本较高
